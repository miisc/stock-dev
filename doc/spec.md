# 股票回测系统项目范围 (Spec)

## 项目概述

### 项目目标
开发一个基于Python的股票回测系统，专注于A股市场的策略回测和分析。该系统旨在为量化交易初学者和研究人员提供一个简单易用的回测平台，帮助他们验证交易策略的有效性。

### 项目背景
在量化交易领域，回测是验证策略有效性的关键步骤。目前市面上虽然有一些成熟的回测平台，但大多功能复杂、学习成本高，或者需要付费使用。本项目旨在构建一个开源、轻量级、易于理解和使用的回测系统，特别适合：
- 量化交易初学者学习回测概念
- 研究人员快速验证策略想法
- 教育机构用于教学演示

### 核心价值主张
1. **简单易用**: 提供直观的用户界面，降低使用门槛
2. **教育导向**: 代码结构清晰，便于学习和理解
3. **快速验证**: 支持快速策略回测，提高研究效率
4. **开源免费**: 完全开源，无使用限制

## 决策与权衡

### 技术选型决策

#### 数据源选择：AKShare
**决策**: 选择AKShare作为主要数据源
**理由**: 
- 免费开源，无API调用限制
- 数据覆盖面广，支持A股历史数据
- 社区活跃，维护及时
- Python原生支持，集成简单

**权衡**: 
- 优势：免费、易用、数据质量较好
- 劣势：相比付费数据源，数据更新可能稍有延迟
- 缓解措施：实现数据缓存机制，减少实时依赖

#### 数据库选择：SQLite
**决策**: 选择SQLite作为数据存储方案
**理由**:
- 轻量级，无需额外安装配置
- 适合中小规模数据存储
- Python内置支持，部署简单
- 文件型数据库，便于备份和迁移

**权衡**:
- 优势：零配置、便携性好、适合单机使用
- 劣势：并发写入性能有限，不适合大规模分布式应用
- 缓解措施：本项目主要面向单用户回测场景，SQLite完全满足需求

#### 用户界面选择：Streamlit
**决策**: 选择Streamlit作为Web界面框架
**理由**:
- 专为数据科学应用设计
- 代码简洁，开发效率高
- 内置图表组件，适合展示回测结果
- 学习成本低，便于维护

**权衡**:
- 优势：快速开发、美观界面、数据可视化友好
- 劣势：自定义能力有限，不适合复杂交互
- 缓解措施：本项目界面需求相对简单，Streamlit完全满足

### 功能范围决策

#### 策略复杂度：基础技术指标策略
**决策**: 专注于基础技术指标策略，不包含复杂机器学习策略
**理由**:
- 基础策略更容易理解和学习
- 计算简单，性能开销小
- 适合教学和演示场景
- 为后续扩展奠定基础

**权衡**:
- 优势：简单易懂、性能好、教育价值高
- 劣势：策略覆盖面有限，不适合高级用户
- 缓解措施：预留策略接口，支持用户自定义扩展

#### 数据频率：日线数据
**决策**: 仅支持日线数据回测，不支持分钟级或tick级数据
**理由**:
- 日线数据更容易获取和处理
- 适合中长期策略回测
- 降低系统复杂度和存储需求
- 符合初学者学习需求

**权衡**:
- 优势：数据获取简单、处理效率高、存储需求小
- 劣势：无法支持高频策略回测
- 缓解措施：明确项目定位，专注于中长期策略

## 项目范围

### 包含范围

#### 核心功能模块
1. **数据获取模块**
   - 从AKShare获取A股日线数据
   - 数据清洗和预处理
   - 本地SQLite存储

2. **策略框架模块**
   - 策略基类定义
   - 双均线策略实现
   - 布林带策略实现
   - RSI策略实现
   - 策略参数配置

3. **回测引擎模块**
   - 前向遍历回测逻辑
   - 交易执行模拟
   - 交易成本计算
   - 仓位管理

4. **评估分析模块**
   - 绩效指标计算
   - 结果可视化
   - 交易记录导出

5. **用户界面模块**
   - 策略配置界面
   - 回测执行控制
   - 结果展示界面

#### 支持功能
- 错误处理和日志记录
- 配置文件管理
- 单元测试覆盖
- 基础文档

### 明确排除范围

#### 功能排除
1. **多资产类别支持**
   - 仅支持A股，不支持港股、美股、期货、外汇等
   - 理由：简化系统复杂度，专注单一市场

2. **高频交易支持**
   - 仅支持日线数据，不支持分钟级或tick级数据
   - 理由：降低系统复杂度和存储需求

3. **实时交易功能**
   - 仅支持历史回测，不支持实时交易
   - 理由：避免监管风险，专注回测功能

4. **复杂衍生品支持**
   - 不支持期权、期货等衍生品回测
   - 理由：简化系统复杂度

5. **高级机器学习策略**
   - 不内置复杂机器学习策略
   - 理由：保持系统简单性，专注基础策略

#### 技术排除
1. **分布式架构**
   - 采用单机架构，不支持分布式部署
   - 理由：简化部署和维护

2. **实时数据流**
   - 不支持实时数据推送和处理
   - 理由：专注历史回测场景

3. **复杂用户管理**
   - 不支持多用户、权限管理等复杂功能
   - 理由：简化系统设计

## 风险与缓解措施

### 主要风险
1. **数据源稳定性风险**
   - 风险：AKShare API可能不稳定或变更
   - 影响：数据获取失败
   - 缓解：实现重试机制，准备备用数据源

2. **性能风险**
   - 风险：大量数据回测可能超时
   - 影响：用户体验差
   - 缓解：数据库优化，算法优化

3. **需求变更风险**
   - 风险：开发过程中需求可能变更
   - 影响：进度延期
   - 缓解：明确范围，变更控制流程

### 应急计划
1. **数据源问题**
   - 短期：使用本地缓存数据
   - 长期：集成其他免费数据源

2. **性能问题**
   - 短期：优化算法，减少数据量
   - 长期：考虑升级数据库或架构

3. **进度延期**
   - 短期：调整功能优先级
   - 长期：增加资源或延长周期
